1. Problem Statement & Current State

An engineering and branding audit of the codebase reveals that the project currently uses multiple inconsistent representations of the company brand name across the application.

Specifically, textual inspection across layout headers, metadata definitions, and user interface components identified three distinct variations:

  • "Edfoal"
  • "EdFoal"
  • "EdFoal AI"

At present, there is no centralized source of truth for the brand name. Instead, string literals are hardcoded independently across layout files, SEO metadata declarations, and UI components. This decentralization has resulted in noticeable branding discrepancies between page headers, document titles, promotional callouts, and footer sections.

2. Root Cause Analysis (Business & Engineering)

To provide a comprehensive assessment for project stakeholders, the root cause of this inconsistency has been evaluated from both business governance and software engineering perspectives.

Business Root Cause

From an organizational and process standpoint, the inconsistency stems from a lack of formal branding governance during the development lifecycle:

  • Lack of Formal Branding Governance: There was no mandatory brand style guide enforced during the copywriting and UI prototyping phases.
  • Absence of Centralized Documentation: Contributors lacked an authoritative reference document specifying the canonical capitalization and spacing of the brand name.
  • Independent Content Evolution: Different developers and content creators worked on separate pages and features asynchronously, leading to divergent spellings and naming conventions over time.
  • Unenforced Guidelines: Even where brand preferences existed informally, they were not communicated as a strict requirement to external contractors or engineering teams.

Engineering Root Cause

From a technical and architectural perspective, the issue is a classic violation of modular design and configuration management principles:

  • Hardcoded String Literals: The brand name is manually typed as a raw string across dozens of individual component files rather than being referenced from a shared configuration variable.
  • Absence of a Global Constant: No global BRAND_NAME constant or configuration export was established during the initial project scaffolding.
  • Copy-Paste Prototyping: Developers copied text directly from design mockups or previous components without standardizing the underlying data structures.
  • Scope of Code Reviews: Pull request reviews and technical evaluations focused primarily on functional correctness, responsive layout, and runtime performance rather than textual or branding consistency.
  • Lack of Automated Validation: The repository lacks automated linting rules, custom static analysis checks, or pre-commit hooks to detect and flag non-canonical brand strings.

3. Business & Technical Impact

Although categorized as a low-severity finding due to the lack of runtime functional impairment, the ongoing presence of inconsistent branding introduces tangible risks and inefficiencies.

Business Impact

  • Reduced Brand Professionalism: Presenting variations such as "Edfoal" and "EdFoal" on the same website diminishes the perceived maturity and attention to detail of the organization.
  • Inconsistent User Experience: Visitors navigating across different sections of the platform may experience subtle confusion regarding the official identity of the company and its product suite.
  • Weakened Brand Identity: Inconsistent capitalization dilutes brand recognition in competitive markets and digital advertising channels.
  • Marketing Asset Inconsistency: External promotional materials, social media share cards (OpenGraph tags), and search engine search results display conflicting brand names.

Technical Impact

  • High Refactoring Overhead: Modifying or rebranding the company name currently requires manual edits across multiple disparate files rather than a single line change.
  • Increased Maintenance Effort: Future developers must expend cognitive effort deciding which spelling to use when creating new pages or components.
  • Regression Risk: Manual multi-file updates carry a high probability of human error, increasing the likelihood that certain instances will be overlooked during future updates.
  • Violation of Core Engineering Principles: Hardcoding shared domain concepts violates the DRY (Don't Repeat Yourself) and Single Source of Truth principles, encouraging further technical debt and data duplication.

4. Recommended Remediation

We recommend implementing a centralized branding configuration strategy to eliminate redundancy and enforce canonical brand consistency across the entire application.

Step 1

Create a single source of truth within the shared utilities or constants module.

Example modification in src/lib/constants.ts:

export const BRAND_NAME = "EdFoal";

Step 2

Replace every hardcoded occurrence of the brand string across the codebase ("Edfoal", "EdFoal", and "EdFoal AI") with a programmatically imported reference:

import { BRAND_NAME } from "@/lib/constants";

Use BRAND_NAME wherever the canonical company name is required in JSX, layout headers, and text blocks.

Step 3

If a marketing page or service offering intentionally uses an expanded product title such as "EdFoal AI", this string must be derived programmatically from the canonical constant rather than manually typed as a raw string.

Example derivation:

const productTitle = `${BRAND_NAME} AI`;

Step 4

Systematically update all document metadata declarations, navigation bars, footer components, layout wrappers, SEO tags, and OpenGraph page titles to consume the centralized constant from src/lib/constants.ts.

5. Verification Plan

To confirm the successful execution of the remediation plan and prevent future regression, the following verification protocol must be executed:

  1. Repository-Wide Static Scan: Perform a comprehensive regular expression and text search across the entire project repository (src/) for string literals matching:
    • Edfoal
    • EdFoal
    • EdFoal AI
  2. Single Source of Truth Audit: Confirm that only src/lib/constants.ts (or the designated configuration file) contains the raw canonical brand string literal.
  3. Static Analysis & Linting: Execute the automated linting suite to ensure no syntax errors or unused import warnings were introduced: 
    npm run lint
  4. Production Bundle Verification: Execute the production build compiler to verify that template literals and constant imports compile cleanly into static and server-rendered routes: 
    npm run build
  5. Visual & SEO Inspection: Perform manual verification across browser endpoints and view-source metadata to confirm that the brand name renders consistently across all visual layouts and search engine tags.

6. Conclusion

Finding RCA-2.5 represents a low-severity architectural and operational finding with zero impact on runtime software functionality or system availability. However, resolving this discrepancy is a high-value structural optimization. By establishing a Single Source of Truth for organizational identity, the engineering team directly enhances brand professionalism, eliminates maintenance friction, and reinforces long-term code quality across the repository.
