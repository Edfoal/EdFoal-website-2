1. Problem Statement & Current State

The project's dependency manifest (package.json) currently installs two redundant copies of the same animation library ecosystem:

  • framer-motion: Installed at version ^12.40.0 (used across 40+ files).
  • motion: Installed at version ^12.40.0 (used in 5 files: OriginButton.tsx, LayoutGrid.tsx, LargeNameFooter.tsx, CardSpotlight.tsx, and animated-dock.tsx).

While motion (specifically motion/react) is the newly rebranded NPM package name for framer-motion v12+, having both packages installed simultaneously creates dependency duplication and confusion regarding which import path developers should use.

2. Root Cause Analysis

⚠ Third-Party Component Copy-Pasting & Rebranding Transition
The root cause is the historical rebranding of framer-motion to motion by Motion (formerly Framer). When UI developers integrated third-party component libraries (such as Aceternity UI or custom animated primitives), some newly copied components imported from motion/react while the core codebase was built using framer-motion. Instead of aligning import paths, npm install motion was executed, adding a second redundant dependency to package.json.
  1. Inconsistent Import Standards: Lack of an ESLint rule or project guideline specifying a single canonical animation import path.
  2. Rebranding Overlap: framer-motion v12 and motion v12 share identical underlying APIs and rendering engines, making the duplication silent during compilation.

3. Business & Technical Impact

  • Bundle & Node_Modules Bloat: Installing both packages duplicates dependency trees in node_modules, increasing npm install times and CI/CD build overhead.
  • Potential Version Drift & Bundle Splitting: If the two packages ever drift in version (e.g., framer-motion@12.40.0 vs motion@12.45.0), Webpack/Turbopack may bundle two independent animation runtime engines into the client JavaScript bundle, significantly inflating page weight and degrading page load performance (Core Web Vitals).
  • Developer Cognitive Overhead: Future contributors face confusion when writing new animations, unsure whether to import motion from "framer-motion" or "motion/react".

4. Remediation Plan

We propose standardizing on framer-motion, as it represents >85% of existing codebase usage (40+ files vs 5 files):

  1. Standardize Imports in the 5 Affected Files: Refactor the 5 components (OriginButton.tsx, LayoutGrid.tsx, LargeNameFooter.tsx, CardSpotlight.tsx, and animated-dock.tsx) to import from "framer-motion" instead of "motion/react" or "motion".
  2. Prune Redundant Dependency from package.json: Remove "motion": "^12.40.0" from dependencies and run npm uninstall motion.

5. Verification Plan

  1. Import Scan: Execute grep search across src/ for from "motion to confirm zero remaining imports from the deprecated package name.
  2. Dependency Check: Verify package.json only lists "framer-motion".
  3. Production Build Test: Execute npm run build to verify that all 5 refactored components compile and animate cleanly with framer-motion.
