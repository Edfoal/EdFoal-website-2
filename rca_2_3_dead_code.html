1. Problem Statement & Current State

The codebase currently contains several dead code files and duplicated datasets across src/lib/ and src/data/:

  1. Unused API Fetch Client (src/lib/api.ts:1-26): Defines a generic apiClient<T> wrapper around fetch(). Static analysis confirms this client is never imported or executed anywhere in the project.
  2. Unused Site Constants (src/lib/constants.ts:1-11): Defines SITE_METADATA and SOCIAL_LINKS dictionaries. These constants are never imported; site titles, SEO metadata, and social URLs are hardcoded directly inside layout files.
  3. Duplicated Services Dataset (src/data/services.ts vs GetStartedModal.tsx:17): The application defines service offerings in two independent places:
    • src/data/services.ts: Exported servicesData array.
    • GetStartedModal.tsx:17: An inline, hardcoded servicesData array containing duplicated descriptions and image URLs.
  4. Unused Testimonials Data (src/data/testimonials.ts): Defines an exported testimonialSlides array that is never imported by any active carousel or section component.

2. Root Cause Analysis

⚠ Decoupled Architecture & Prototype Residue
During early project setup, standard utility files (api.ts, constants.ts, testimonials.ts) were scaffolded based on Next.js starter templates. As feature development progressed, UI developers opted to hardcode data inline (such as inside GetStartedModal.tsx and layout headers/footers) for rapid visual iterations rather than importing from centralized data sources.
  1. Lack of Single Source of Truth Strategy: Teams developed feature modals and layout sections independently without enforcing centralized data schemas.
  2. Unpruned Boilerplate: Starter utilities and prototype data arrays were never garbage-collected after production features were finalized.

3. Business & Technical Impact

  • Data Drift & Brand Inconsistency: Having service descriptions and titles duplicated across services.ts and GetStartedModal.tsx guarantees that future content updates will fall out of sync (e.g., updating a pricing tier or feature bullet in one file but forgetting the other).
  • SEO & Social Drift: Hardcoding metadata and social URLs across layouts instead of using constants.ts makes global branding updates error-prone.
  • Bundle Bloat & Cognitive Overhead: Developers investigating networking or data flows waste time analyzing dead code utilities like apiClient that do not reflect actual runtime behavior.

4. Remediation Plan

We propose a clean structural cleanup and consolidation:

  1. Centralize Services Data (Single Source of Truth):
    • Consolidate all rich service details into src/data/services.ts.
    • Refactor GetStartedModal.tsx to import and consume servicesData from @/data/services, deleting the inline hardcoded array at line 17.
  2. Wire Up or Prune Constants: Import SITE_METADATA and SOCIAL_LINKS into layout and footer components, or delete constants.ts if hardcoded layouts are preferred.
  3. Delete Unused Dead Code Files: Delete src/lib/api.ts and src/data/testimonials.ts.

5. Verification Plan

  1. Static Reference Check: Confirm via grep search that zero references remain to deleted dead files and zero inline servicesData duplicates exist.
  2. Production Build Test: Run npm run build to verify that removing dead files and consolidating service imports compiles cleanly without TypeScript or routing errors.
