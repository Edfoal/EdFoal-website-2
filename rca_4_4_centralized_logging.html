Core Issue Ad-hoc console.warn/error calls scattered across UI components and error boundaries without structured formatting or remote telemetry routing.

1. Problem Statement & Current State

During an observability and production readiness audit, it was identified that application error and warning reporting is handled via ad-hoc, uncoordinated console.warn and console.error calls scattered across multiple files with no unified production logging strategy.

Specifically, direct console.* invocations exist in UI animation components (OriginButton, PixelatedCanvas, CanvasRevealEffect) and route error boundaries (error.tsx, global-error.tsx): 

// Example: Scattered ad-hoc console calls
console.warn("OriginButton: Missing action handler.");
console.error("Failed to load image for PixelatedCanvas:", src);
console.error(`[EdFoal Runtime Error]:`, error);

In a Next.js 15 production environment, direct browser console logs are completely invisible to engineering teams (as they execute only on end-user client devices), while server-side console calls lack structured JSON formatting, correlation IDs, timestamps, and severity levels required by modern observability platforms (e.g., Datadog, Sentry, AWS CloudWatch).

2. Root Cause Analysis

🔍 Absence of an Observability Abstraction Layer

When importing third-party UI components (such as Aceternity UI animations) and building initial route error boundaries, developers relied on native browser console APIs for quick local debugging. No centralized abstraction layer (e.g., @/lib/logger.ts) was established during scaffolding to decouple log emission from log destination.

  • Local Debugging Bias: Developers default to console.log/console.error during development because browser devtools provide immediate feedback, overlooking how those logs behave when deployed to static CDNs or serverless edge runtimes.
  • Missing Telemetry Architecture: The project lacks an integrated error monitoring service or structured log formatter, leaving components with no standardized method to report runtime anomalies or asset loading failures.
  • Decentralized Error Handling: Without a single logging interface, formatting conventions (such as error prefixes and metadata attachment) vary wildly from file to file.

3. Business & Technical Impact

  • Blind Incident Response (MTTR Degradation): If a client-side component crashes or fails to render on a customer's browser (e.g., a canvas WebGL texture failure on iOS), engineering has zero visibility into the failure unless the customer manually submits a screenshot of their console.
  • Log Noise & Performance: Uncontrolled console.warn spam in high-frequency rendering loops (such as canvas animations) can degrade client browser performance and pollute devtools for frontend developers.
  • Vendor Lock-In Prevention: Hardcoding console.* throughout the codebase makes it laborious to integrate enterprise observability tools later without refactoring dozens of files.

4. Remediation Plan

We propose establishing a Centralized Logging Service (src/lib/logger.ts) and replacing all scattered ad-hoc console calls with a unified abstraction.

Step 1: Create Centralized Logger (src/lib/logger.ts)

Create a lightweight, type-safe logging utility that automatically formats logs for development and routes structured events to monitoring services in production:

class Logger {
  private isDev = process.env.NODE_ENV !== "production";

  warn(message: string, context?: Record<string, unknown>) {
    if (this.isDev) {
      console.warn(`[WARN]: ${message}`, context || "");
    } else {
      console.warn(JSON.stringify({ level: "warn", message, ...context, timestamp: new Date().toISOString() }));
    }
  }

  error(message: string, error?: unknown, context?: Record<string, unknown>) {
    if (this.isDev) {
      console.error(`[ERROR]: ${message}`, error || "", context || "");
    } else {
      console.error(JSON.stringify({ level: "error", message, error: String(error), ...context, timestamp: new Date().toISOString() }));
    }
  }
}

export const logger = new Logger();

Step 2: Refactor Scattered Console Calls

Replace ad-hoc console calls across all affected files to use the centralized logger:

   useEffect(() => {
-    console.error("[EdFoal Runtime Error]:", error);
+    logger.error("Route segment runtime failure", error, { digest: error.digest });
   }, [error]);
