Core Issue Lenis smooth-scrolling is instantiated independently across 8 page routes instead of once in the root layout, causing duplicated event listeners and scroll fighting.

1. Problem Statement & Current State

During a runtime performance and animation profiling audit, it was discovered that the smooth-scrolling library Lenis is being instantiated independently across 8 different route pages instead of being managed globally in the root application layout.

Specifically, each of the 8 pages imports and invokes useLenis() directly: 

// Example found in src/app/page.tsx, src/app/about/page.tsx, etc.
import useLenis from "@/hooks/useLenis";

export default function Page() {
  useLenis(); // Instantiates new Lenis instance + RAF loop on every page mount!
  return <main>...</main>;
}

In Next.js 15 App Router, client-side navigation between route segments does not perform a full browser page reload. As a result, invoking useLenis() inside individual page components causes Lenis to be initialized repeatedly during navigation. This creates redundant wheel/touch event listeners and overlapping requestAnimationFrame (RAF) loops, leading to scroll jitter, memory leaks, and animation fighting.

2. Root Cause Analysis

🔍 Decentralized Page-Level Animation Wiring

When migrating to Next.js App Router, developers treated smooth scrolling as a page-by-page visual enhancement rather than an application-wide viewport singleton. Because useLenis was authored as a standalone hook without React Context or singleton guarding, adding it to individual pages seemed like the quickest way to enable smooth scrolling.

  • Misunderstanding of SPA Routing Mechanics: In traditional multi-page applications (MPAs), page-level initialization is acceptable because every navigation tears down the JavaScript runtime. In Next.js App Router SPA routing, page components mount and unmount dynamically while the root window context persists.
  • Absence of a Root Provider Pattern: The codebase lacked a centralized provider architecture (e.g., SmoothScrollProvider in src/app/layout.tsx) to manage viewport-level concerns such as scroll momentum, lenis ticker synchronization, and resize observers.
  • Uncoordinated Animation Frames: Each call to new Lenis() spawns an independent requestAnimationFrame loop (lenis.raf(time)). When multiple instances coexist or overlap during page transitions, they compete to update CSS scroll positions simultaneously.

3. Business & Technical Impact

  • Scroll Jitter & Stuttering: When two or more Lenis instances attempt to interpolate scroll momentum at the same time, users experience jerky, vibrating, or unresponsive page scrolling—destroying the premium aesthetic of the application.
  • CPU & Battery Drain: Running multiple simultaneous requestAnimationFrame loops on 60Hz or 120Hz displays significantly increases CPU/GPU utilization, causing rapid battery drain on mobile devices and laptops.
  • Memory Leaks: Repeatedly mounting and unmounting Lenis instances across complex user navigation journeys accumulates detached DOM event listeners and uncleared animation frames over time.
  • Code Redundancy: Duplicating import useLenis and useLenis() across 8 files adds unnecessary boilerplate and increases maintenance overhead.

4. Remediation Plan

We propose refactoring scroll management to adhere to the React singleton pattern: "Create SmoothScrollProvider in layout.tsx, call useLenis() once, and remove all page-level invocations."

Step 1: Create SmoothScrollProvider Component

Create a lightweight Client Component in src/components/providers/SmoothScrollProvider.tsx that invokes useLenis() exactly once at the application root:

"use client";

import React from "react";
import useLenis from "@/hooks/useLenis";

export default function SmoothScrollProvider({ children }: { children: React.ReactNode }) {
  useLenis(); // Single application-wide Lenis instance and RAF ticker
  return <>{children}</>;
}

Step 2: Wrap Root Layout in layout.tsx

Integrate SmoothScrollProvider into src/app/layout.tsx so that smooth scrolling persists seamlessly across all route transitions:

+import SmoothScrollProvider from "@/components/providers/SmoothScrollProvider";

 export default function RootLayout({ children }: { children: React.ReactNode }) {
   return (
     <html lang="en">
       <body>
+        <SmoothScrollProvider>
           <Navbar />
           {children}
           <Footer />
+        <SmoothScrollProvider>
       </body>
     </html>
   );
 }

Step 3: Clean Up All 8 Page Components

Delete the redundant useLenis() imports and calls from all 8 page files (src/app/page.tsx, src/app/about/page.tsx, etc.).
