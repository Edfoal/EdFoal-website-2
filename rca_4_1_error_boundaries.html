1. Problem Statement & Current State

The Next.js App Router application currently lacks root or route-level error boundaries (src/app/error.tsx and src/app/global-error.tsx).

In Next.js 15 / React 19, when an unhandled runtime exception occurs during rendering or inside a Client Component (such as a null pointer exception, animation state crash, or failed third-party script hydration), React unmounts the broken component tree. Without a route segment error boundary to intercept and contain the failure, the entire application collapses into a completely blank White Screen of Death (WSOD): 

[Runtime Exception in Client Component] ──> No <ErrorBoundary> ──> Whole App Unmounts (White Screen)

Furthermore, without defensive error boundaries, client-side crashes cannot be programmatically caught, logged, or forwarded to telemetry services (such as Sentry, Datadog, or custom analytics), leaving engineering teams blind to critical production outages.

2. Root Cause Analysis

⚠️ App Router Defensive Routing Gap

During initial project development and migration to Next.js App Router (src/app/), engineering efforts focused heavily on happy-path page construction (page.tsx, layout.tsx) and UI presentation, while file-system routing conventions for fault tolerance (error.tsx, global-error.tsx) were omitted.

  • Server vs. Client Catch-Boundary Miscomprehension: In App Router, standard try/catch blocks inside async Server Components only catch server-side data fetching errors. They cannot catch React client-side rendering exceptions, event handler crashes, or browser hydration failures. Catching client component errors strictly requires Next.js error.tsx conventions (which automatically wrap route segments in React <ErrorBoundary> primitives under the hood).
  • Missing Root Safeguard: Even if individual components handle their own state, the absence of src/app/global-error.tsx means that if the root layout.tsx (or top-level theme/font providers) fails to render, there is zero fallback UI, resulting in a fatal browser crash.
  • Lack of Architecture Enforcement: No automated linter rule or CI/CD architecture gate was configured to enforce the presence of mandatory error boundaries in root route segments.

3. Business & Technical Impact

  • Catastrophic User Experience (High Bounce Rate): A single unhandled TypeError in an interactive UI element (such as the 3D Globe, Framer Motion animations, or navigation menus) crashes the entire page for the visitor. Users experience a frozen or blank white screen with no way to recover or navigate back, destroying brand credibility.
  • Zero Observability & Silent Outages: Because errors bypass structured logging, critical production crashes fail silently on user devices without alerting the engineering team, drastically increasing Mean Time to Detection (MTTD) and Mean Time to Resolution (MTTR).
  • Broken Conversion Funnels: If a crash occurs during interactive lead generation workflows (e.g., Contact Form submission or Get Started modals), potential clients are lost permanently without any error feedback or retry mechanism.

4. Remediation Plan

We propose implementing the canonical Next.js App Router fault-tolerance architecture immediately by creating two defensive error boundary files styled to match EdFoal's premium cosmic design system.

Step 1: Create Route Segment Error Boundary (src/app/error.tsx)

This file intercepts crashes in all nested pages and layouts, presenting a branded recovery interface with a reset() button to re-render the segment without doing a hard browser refresh.

+"use client";
+
+import React, { useEffect } from "react";
+import { AlertTriangle, RefreshCw, Home } from "lucide-react";
+import Link from "next/link";
+import { BRAND_NAME } from "@/lib/constants";
+
+export default function ErrorBoundary({ error, reset }: { error: Error & { digest?: string }; reset: () => void; }) {
+  useEffect(() => {
+    console.error(`[${BRAND_NAME} Runtime Error]:`, error);
+  }, [error]);
+
+  return (
+    <div className="relative min-h-[70vh] w-full flex flex-col items-center justify-center px-6 text-center bg-black text-white overflow-hidden select-none">
+      <div className="relative z-10 flex flex-col items-center max-w-md mx-auto p-8 rounded-2xl border border-white/10 bg-zinc-900/50 backdrop-blur-xl shadow-2xl">
+        <h2 className="text-2xl font-bold tracking-tight text-white mb-2">Something went wrong</h2>
+        <p className="text-sm text-zinc-400 mb-8">An unexpected error occurred. Our engineering team has been notified.</p>
+        <button onClick={reset} className="px-6 py-3 rounded-xl bg-blue-600 hover:bg-blue-500 text-white font-semibold text-sm">Try Again</button>
+      </div>
+    </div>
+  );
+}

Step 2: Create Root Layout Error Boundary (src/app/global-error.tsx)

This file wraps the entire application. If the root layout.tsx fails, global-error.tsx replaces it entirely and must define its own <html> and <body> tags.

+"use client";
+
+import React, { useEffect } from "react";
+import { BRAND_NAME } from "@/lib/constants";
+
+export default function GlobalError({ error, reset }: { error: Error & { digest?: string }; reset: () => void; }) {
+  useEffect(() => {
+    console.error(`[${BRAND_NAME} Fatal Global Error]:`, error);
+  }, [error]);
+
+  return (
+    <html lang="en">
+      <body className="min-h-screen bg-[#030914] text-white flex flex-col items-center justify-center p-6 text-center font-sans">
+        <h1 className="text-2xl font-bold text-red-400 mb-3">Fatal System Error</h1>
+        <button onClick={reset} className="px-6 py-3 rounded-xl bg-blue-600 hover:bg-blue-500 text-white font-semibold text-sm">Reload Application</button>
+      </body>
+    </html>
+  );
+}
