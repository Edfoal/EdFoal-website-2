1. Problem Statement & Current State

During an architecture and security audit of core utility libraries, a latent information leakage vulnerability was identified in the generic fetch client src/lib/api.ts between lines 18 and 22.

When an HTTP request returns a non-200 status code (!response.ok), the client executes await response.text() and directly injects the raw server response body into a thrown JavaScript Error object: 

  if (!response.ok) {
    const errorBody = await response.text();
    throw new Error(
      `API error [${response.status}]: ${errorBody || response.statusText}`
    );
  }

If a backend server or upstream API returns an unhandled 500 Internal Server Error, database query failure, or stack trace, throwing the raw errorBody directly to the caller risks leaking sensitive internal backend details (such as SQL syntax, table names, file system paths, or internal microservice IPs) to the client browser or frontend error reporting UI.

Although src/lib/api.ts is currently dead code (not active in any production page routing today), leaving raw response forwarding in a core library represents an immediate latent hazard as soon as future feature development adopts this utility.

2. Root Cause Analysis

🔍 Naïve Error Handling in Utility Scaffold

During initial project scaffolding, src/lib/api.ts was authored as a quick-start HTTP wrapper. The developer prioritized immediate debugging convenience—dumping whatever text the backend returned into the error message—without implementing data sanitization or separation between server logging and client-facing error messages.

  • Lack of Boundary Sanitization: The utility assumes all API error responses are safe, human-readable strings intended for end users, ignoring the fact that upstream web servers often dump raw HTML stack traces or database exception details during unhandled crashes.
  • Missing Structured Error Schema: The application lacks a standardized error schema (e.g., { code: string; message: string }) for API communications, forcing the client to treat arbitrary text payloads as valid error messages.
  • Dead Code Stagnation: Because the function is not actively invoked in the current codebase, standard QA and penetration testing did not trigger or expose the vulnerability during runtime testing.

3. Business & Technical Impact

  • Information Leakage & Attack Surface Expansion: If future developers wire this client to a backend authentication or billing endpoint, raw server exceptions (e.g., PGError: relation "users" does not exist) could be exposed in browser devtools or user error modals, providing malicious actors with valuable reconnaissance data.
  • Unprofessional UI Degradation: If an upstream server returns a custom HTML error page (such as a Cloudflare or Nginx 502 Bad Gateway HTML dump), throwing raw text causes ugly HTML markup strings to render inside frontend alert toasts or error boundaries.
  • Engineering Rigor: Proactively hardening utility libraries before they are wired into active feature flows prevents technical debt and security regressions.

4. Remediation Plan

We propose hardening src/lib/api.ts by adhering to the defensive rule: "Parse structured errors; never forward raw server responses."

Step 1: Harden apiClient in src/lib/api.ts

Modify lines 18–23 to safely parse structured JSON error messages if provided by the backend, while defaulting to a clean, sanitized HTTP status message for all unhandled text or HTML responses:

   if (!response.ok) {
-    const errorBody = await response.text();
-    throw new Error(
-      `API error [${response.status}]: ${errorBody || response.statusText}`
-    );
+    let errorMessage = `Request failed with status ${response.status} (${response.statusText})`;
+    
+    try {
+      const errorJson = await response.json();
+      if (errorJson && typeof errorJson.message === "string") {
+        errorMessage = errorJson.message;
+      }
+    } catch {
+      // Fallback: If response is not JSON (e.g. HTML stack trace), do NOT leak raw body to client
+    }
+
+    throw new Error(errorMessage);
   }
